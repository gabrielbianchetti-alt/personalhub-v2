# Backlog — Análise de otimização (09/jul/2026)

Workflow de 6 dimensões com verificação adversarial: 40 achados confirmados, 3 parciais, 0 refutados.

## Confirmados
### [perf-servidor-1] auth.getUser() (round-trip de rede) serializado no início de TODA server action e do recibo — proxy já valida local com getClaims
- **Onde:** src/app/actions/registros.ts:32 · severidade media · esforço baixo
- **Evidência:** registros.ts:30-33 `const { data: { user } } = await supabase.auth.getUser();` dentro de validar(), executado antes de qualquer query em marcarFalta/desmarcarFalta/adicionarExtra/removerExtra/confirmarDia/resolverDiaComFaltas. O mesmo padrão em professorAtual() de actions/cobranca.ts:16-19, actions/pacote.ts:16-19, actions/alunos.ts:70-73, actions/conta.ts:10-13 e no recibo (src/app/recibo/[alunoId]/route.tsx:27-30, onde o getUser bloqueia ANTES do Promise.all das queries). Em contraste, o proxy (src/lib/supabase/proxy.ts:51) já usa `supabase.auth.getClaims()` com o comentário 'valida o token LOCALMENTE — sem ida à rede em toda request'.
- **Impacto:** Cada toque do professor no celular (Faltou, +extra, marcar enviado/pago, vender pacote) paga um round-trip HTTP ao Auth do Supabase (~100-300ms saindo da Vercel) em série, antes da query de verdade — e o request já passou pelo proxy, que acabou de validar a mesma sessão. No caminho do dinheiro (marcarEnviado/marcarPago) isso soma à cadeia serial existente. Latência percebida de toda interação praticamente dobra o RTT de banco.
- **Sugestão:** Nas actions e no recibo, trocar `supabase.auth.getUser()` por `supabase.auth.getClaims()` (mesmo racional do proxy — chaves assimétricas validam local): `const { data } = await supabase.auth.getClaims(); const professorId = data?.claims.sub; if (!professorId) throw new Error('Sessão expirada — entre de novo.')`. RLS continua sendo a barreira real; o getClaims só fornece o professor_id e o 401 amigável sem custo de rede.

### [perf-servidor-2] Cobrança: até 4 round-trips seriais — a query de suspensos não depende da cadeia de créditos e as 2 queries de créditos podem ser paralelas entre si
- **Onde:** src/app/cobranca/page.tsx:154 · severidade media · esforço baixo
- **Evidência:** Depois do Promise.all inicial (linhas 46-62), o fluxo é serial: linha 102-108 `await supabase.from('pacotes_creditos')...in('aluno_id', creditoIds)`; linha 117-119 `await buscaAulasPacote(supabase, { de: compras[0] })`; e linhas 153-159 `if (suspensosIds.length > 0) { const { data: susp } = await supabase.from('alunos')...in('id', suspensosIds) }`. `suspensosIds` deriva só de `fechamentos` e `ativos` (linhas 151-152), ambos já resolvidos no primeiro Promise.all — a query de suspensos não precisa esperar a cadeia de créditos. E a query de aulas só usa `compras[0]` como filtro `gte`; buscando por `.in('aluno_id', creditoIds).eq('tipo','aula')` (coberta pelo índice registros_aluno_tipo_data_idx da 0011) e filtrando por compraIso em JS (o que a linha 134 já faz), ela deixa de depender da query de pacotes.
- **Impacto:** Para um professor com alunos de crédito E um suspenso com fechamento no mês, o TTFB da tela do dinheiro é 1+1+1+1 = 4 RTTs seriais ao Supabase (~400-800ms só de rede) quando 2 bastariam. É a tela mais crítica do app (fechamento → WhatsApp).
- **Sugestão:** Disparar em paralelo, logo após o primeiro Promise.all: (a) pacotesQ, (b) aulasQ com `.in('aluno_id', creditoIds).eq('tipo','aula')` sem `de` (filtra `data >= compraIso` em JS, filtro que a linha 134 já aplica), (c) suspensosQ. Um único `Promise.all` extra → 2 RTTs no total. Alternativa que elimina a query (c): nested select no fetch de fechamentos, `.select('*, alunos(id, nome, telefone, valor_mensal, ...)')`. Bônus: a query de pacotes hoje traz TODOS os pacotes históricos de cada aluno para usar só o mais recente (linhas 102-114) — cresce sem limite com repetição de compra; vale um comentário/TODO de limitar quando doer.

### [perf-servidor-3] Perfil do aluno: waterfall de 3 awaits seriais (aluno → pacote → aulas / aluno → registros) quando todas as queries derivam só do `id` da rota
- **Onde:** src/app/alunos/[id]/page.tsx:39 · severidade media · esforço medio
- **Evidência:** Linha 25-31: `await supabase.from('alunos')...eq('id', id).maybeSingle()`. Só depois, no braço créditos, linha 39-45: `await supabase.from('pacotes_creditos')...eq('aluno_id', id)...limit(1)` e linha 50: `await buscaAulasPacote(supabase, { alunoId: id, de: compraIso })` — 3 RTTs em série. No braço mensalidade, linha 65: `await buscaRegistros(supabase, { alunoId: id, de: mesRef, ate: sp.iso })` — 2 RTTs em série. Nenhuma dessas queries precisa do resultado da anterior: pacote e registros filtram por `id` da rota; as aulas só usam `compraIso` como corte `gte`, aplicável em JS.
- **Impacto:** Abrir o perfil de um aluno (navegação frequente a partir do roster) custa 3×RTT de banco em série para aluno de pacote (~300-600ms de TTFB só de rede). Com Promise.all cai para 1×RTT.
- **Sugestão:** Um único `Promise.all` com 4 queries: aluno, último pacote, `buscaAulasPacote({ alunoId: id })` sem `de` (filtrar `a.data >= compraIso` em JS depois) e `buscaRegistros({ alunoId: id, de: mesRef, ate: hoje })`. Depois do all, usar só o braço que o `modo_cobranca` pedir — a query especulativa descartada é barata (índices registros_aluno_tipo_data_idx e pacotes_aluno_recente_idx cobrem ambas) e o ganho de latência é de 2 RTTs.

### [perf-servidor-5] marcarPago busca a mesma linha de fechamentos duas vezes na mesma request (snapshotMensalidade + fetch de status), em série
- **Onde:** src/app/actions/cobranca.ts:130 · severidade baixa · esforço baixo
- **Evidência:** snapshotMensalidade já consulta fechamentos nas linhas 41-46: `.from('fechamentos').select('ajuste_manual, ajuste_motivo').eq('aluno_id', alunoId).eq('mes_referencia', mesRef).maybeSingle()`. marcarPago então refaz a MESMA consulta nas linhas 130-135 com outras colunas: `.select('status, aulas_esperadas, faltas, extras, valor_final')` sobre a mesma chave (aluno_id, mes_referencia) — um RTT extra serial entre o snapshot e o upsert.
- **Impacto:** O toque 'marcar pago' (caminho do dinheiro) faz 4 RTTs seriais (getUser + Promise.all do snapshot + fetch de status + upsert) quando 3 bastam. Latência extra de ~100ms por confirmação de pagamento no celular.
- **Sugestão:** Ampliar o select do fechamento dentro de snapshotMensalidade para `'status, aulas_esperadas, faltas, extras, valor_final, ajuste_manual, ajuste_motivo'` e devolver a linha junto do snapshot (ex.: retornar `{ snap, atual }`). marcarPago usa o `atual` retornado em vez de refazer a query; salvarAjuste e marcarEnviado ignoram os campos extras.

### [perf-servidor-6] Políticas RLS usam auth.uid() puro — reavaliado por linha; Supabase recomenda (select auth.uid()) para virar initPlan
- **Onde:** supabase/migrations/0002_rls.sql:30 · severidade baixa · esforço baixo
- **Evidência:** Todas as políticas de 0002_rls.sql usam a forma nua, ex. linhas 30-31: `create policy registros_aula_select on registros_aula for select using (professor_id = auth.uid());` — idem alunos (20-27), pacotes_creditos (40-47), fechamentos (50-57) e dias_resolvidos em 0004 (linhas 25-30). Sem o wrap `(select auth.uid())`, o Postgres chama a função por linha candidata em vez de uma vez por query (lint oficial do Supabase: auth_rls_initplan).
- **Impacto:** registros_aula é a tabela quente: Cobrança varre o mês inteiro de todos os alunos e o saldo de créditos varre desde a compra do pacote. Hoje o overhead é pequeno (centenas de linhas), mas cresce linearmente com o histórico — é custo silencioso em TODA query do app, e a correção é uma migração só de políticas, sem risco de dado.
- **Sugestão:** Migração 0013 recriando as políticas com `using (professor_id = (select auth.uid()))` / `with check (professor_id = (select auth.uid()))` (e `id = (select auth.uid())` em professores). `create or replace`/drop+create é idempotente e não toca dados; o verify-schema.cjs continua válido.

### [perf-cliente-1] CobrancaLista: qualquer estado transitório re-renderiza todos os cards da lista (sem memo, handlers inline)
- **Onde:** src/components/CobrancaLista.tsx:345 · severidade media · esforço medio
- **Evidência:** O item da lista é JSX inline dentro de `{lista.map((item) => { ... })}` (l.345-491), sem componente extraído nem `React.memo` (grep por useMemo/useCallback/memo no src inteiro: zero ocorrências). Todos os estados transitórios vivem no componente-pai: `toast` (setToast l.177 + auto-dismiss no useEffect l.90-94 após 4500ms), `celebrando` (l.159 + clear via setTimeout 1800ms l.160), `festa` (l.138-140), `sheet` (abrir/fechar qualquer sheet, l.429/467/477) e `erro`. Cada um dispara render completo do <ul>, recriando por item os closures `onClick={() => aoPagar(item)}`, `() => aoReabrir(item)`, `() => setSheet({...})` e recomputando `resumoMes(lista)` (l.97 — 6 passadas de filter/reduce) e `diasDesde` por item (l.346).

Com 40-60 alunos (~30 elementos/card com ProgressRing SVG e ícones), cada render reconcilia 1.5-2k elementos num aparelho Android médio — jank perceptível justamente no fechamento do mês, quando o professor encadeia dezenas dessas ações no fluxo guiado.

Com isso toast/celebrando/festa/sheet re-renderizam só o pai + o card afetado. Opcional: `useMemo(() => resumoMes(lista), [lista])`.

### [perf-cliente-3] Barra de progresso 'Recebido' anima width (força layout) em vez de transform
- **Onde:** src/components/CobrancaLista.tsx:317 · severidade baixa · esforço baixo
- **Evidência:** `<div className="h-full rounded-full bg-success transition-[width] duration-500" style={{ width: `${Math.round(pctRecebido * 100)}%` }} />` (l.315-319). `width` é propriedade de layout: durante os 500ms da transição o navegador refaz layout+paint a cada frame, e isso dispara exatamente junto com o render otimista do 'Pago' (a mudança de `pctRecebido`) e o toast — competindo com o trabalho de reconciliação do finding perf-cliente-1 no mesmo frame. O restante do CSS segue a regra transform/opacity (card-in, celebra, shimmer, tab-indicador todos usam transform).
- **Impacto:** Reflow contínuo por 500ms a cada 'Pago' marcado, num momento em que a thread principal já está ocupada re-renderizando a lista. Elemento pequeno, então o custo unitário é baixo — mas é a única animação do app que viola a própria convenção transform/opacity do projeto.
- **Sugestão:** Trocar por transform: filho sempre com `w-full` e `style={{ transform: `scaleX(${pctRecebido})`, transformOrigin: "left" }}` com `transition-transform duration-500`. O container já tem `overflow-hidden rounded-full`, então o visual fica idêntico e a animação vira composited-only. (Não há conflito com a invariante glass/sticky: o card sticky é ancestral, não descendente, do elemento com transform.)

### [robustez-seguranca-1] Erros de negócio das server actions são mascarados em produção — professor vê parágrafo genérico em inglês
- **Onde:** src/app/actions/pacote.ts:56 · severidade alta · esforço medio
- **Evidência:** Todas as actions sinalizam erro esperado com throw: pacote.ts:56 `throw new Error("Pacote esgotado — venda um novo pacote.")`, :67 "Já tem uma aula marcada nesse dia.", registros.ts:27 "Fora da janela de correção (7 dias).", :33 "Sessão expirada — entre de novo.". O client exibe `e.message` (CheckinList.tsx:59, CobrancaLista.tsx:115). Mas o Next mascara erros lançados em Server Actions em produção: node_modules/next/dist/compiled/react-server-dom-turbopack/cjs/react-server-dom-turbopack-client.browser.production.js:1585 `resolveErrorProd()` substitui a mensagem por "An error occurred in the Server Components render. The specific message is omitted in production builds...". A própria doc do Next (dist/docs/01-app/01-getting-started/10-error-handling.md) manda: "For these errors, avoid using try/catch blocks and throw errors. Instead, model expected errors as return values."
- **Impacto:** No deploy da Vercel (produção), NENHUMA das mensagens em pt-BR chega ao professor. Fluxos rotineiros do caminho do dinheiro — pacote esgotado ao agendar aula, aula duplicada no dia, sessão expirada, janela de 7 dias — aparecem como um parágrafo técnico em inglês; a UI otimista desfaz a ação e o professor não tem como saber o que aconteceu nem o que fazer (ex.: precisa vender novo pacote). Em dev tudo funciona, o que esconde o bug até o uso real no celular.
- **Sugestão:** Modelar erro esperado como valor de retorno: actions devolvem `{ ok: true } | { ok: false, erro: string }` e os callers (persistir() em CheckinList/CobrancaLista, AjusteSheet, PacoteSheet, login inline) checam `res.ok` em vez de try/catch. Manter throw só para erro inesperado (aí o mascaramento é desejável). Alternativa mínima: um wrapper `comErro(fn)` nas actions que converte throws conhecidos em retorno serializável.

### [robustez-seguranca-2] Race no +extra/−extra: select-then-update perde incremento e toque duplo estoura erro 23505 não tratado
- **Onde:** src/app/actions/registros.ts:85 · severidade media · esforço medio
- **Evidência:** adicionarExtra lê e depois grava: registros.ts:73-79 seleciona `quantidade`, :85 `update({ quantidade: existente.quantidade + 1 })`. Dois taps rápidos leem quantidade=1 e ambos gravam 2 (incremento perdido). Se a linha ainda não existe, os dois inserem e o segundo recebe 23505, que aqui É lançado (`if (error) throw`, :96) — diferente de marcarFalta, que trata 23505 como idempotente (:51). O botão + não é desabilitado durante o voo: CheckinList.tsx:79-107 aplica otimista e dispara `persistir()` sem lock, então taps consecutivos geram chamadas concorrentes. removerExtra (:113-119) tem o mesmo read-modify-write.
- **Impacto:** Extras contam no fechamento (montaSnapshotFechamento soma excecoes.extras — caminho do dinheiro). Toque duplo no celular: UI mostra 2 extras, banco fica com 1 → cobrança sai menor sem o professor perceber. No caso do insert duplicado, a ação desfaz o otimista e mostra erro (mascarado em produção, ver robustez-seguranca-1).
- **Sugestão:** Tornar o incremento atômico: RPC Postgres `insert ... on conflict (aluno_id, data, tipo) do update set quantidade = registros_aula.quantidade + 1` (e o inverso no decremento). Alternativa sem RPC: no 23505 do insert, refazer como update; no update, usar lock otimista `.eq("quantidade", existente.quantidade)` e re-tentar se 0 linhas. Complementar desabilitando o botão enquanto a transição está pendente.

### [robustez-seguranca-4] Policies de INSERT não validam que aluno_id pertence ao professor — escrita cross-tenant possível e unique global vira vetor de corrupção de contagem
- **Onde:** supabase/migrations/0002_rls.sql:33 · severidade media · esforço baixo
- **Evidência:** 0002_rls.sql:32-33: `create policy registros_aula_insert on registros_aula for insert with check (professor_id = auth.uid())` — checa só professor_id; aluno_id passa por qualquer FK válida (mesmo aluno de OUTRO professor). Idem pacotes_creditos (:42-43) e fechamentos (:52-53). As actions também não conferem posse: registros.ts:43-48 insere `aluno_id` vindo do client direto. O agravante é o unique GLOBAL `unique (aluno_id, data, tipo)` (0001_schema.sql:47): uma linha plantada por outro professor ocupa o slot e o marcarFalta da vítima recebe 23505 — que o app trata como sucesso idempotente (registros.ts:51 `if (error && error.code !== "23505") throw`) — ou seja, a falta da vítima silenciosamente nunca é registrada (a linha alheia é invisível pra ela via RLS de select).
